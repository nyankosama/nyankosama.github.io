高性能Mysql 读书笔记

Mysql概况

Mysql逻辑架构

并发控制

  • 读写锁
  • 锁粒度：表锁：最小锁开销、行锁：最大锁开销，最大并发处理（InnoDB和XtraDB实现）

事务

ACID：

  • 原子性：一个事务具有原子性
  • 一致性：数据库总是从一个一致性的状态装换到另外一个一致性的状态
  • 隔离性：一个事务所做的修改在最终提交之前，对其他的事务是不可见的
  • 持久性：一旦事务提交，则其所做的修改会永久保存到数据库中

隔离级别：

  • READ UNCOMMITTED（未提交读）：事务中的修改，即使没有提交，对其他的事务也是可见的。可造成脏读现象
  • READ COMMITTED（提交读）：大多数数据库的默认隔离级别。一个事务开始时，只能“看见”已经提交的事务所做的修改。但是会出现不可重复读现象，因为无法保证其他的事务的修改情况，因此某一个事务在进行过程中对同一个数据两次读可能不一样，会有脏读现象。
  • REPEATABLE READ（可重复读）：保证同一个事务中多次读取同样的记录结果是一致的。但是可能存在幻读（Phantom Read）现象，即某一个其他事务在某个范围内插入新数据时，如果事务再次读取该范围内的数据会产生幻行。InnoDB和XtraDB通过存储引擎的MVCC（Multiversion Concurrency Control）解决了幻读问题。
  • SERIALIZABLE（可串行化）：最高隔离级别，强制串行执行。

死锁

即两个事务相互占用对方需要的锁。目前有许多死锁检测机制，比如超时，环检测并回滚事务等等

事务日志

大多数存储引擎实现了这个，通常称之为预写式日志，修改数据需要写两次磁盘。

InnoDB MVCC实现

  • 通过在每一行保存两个多于的列
  • 一个记录行创建的时间，一个记录行删除的时间
  • 时间用系统版本号替代，每当开始一个事务的时候，系统版本自动递增

存储引擎

InnoDB

  • 使用MVVC支持高并发
  • 实现四个标准隔离级别
  • 通过间隙锁（next-key locking）策略防止幻读
  • 基于聚簇索引
  • 可预测性读
  • 自适应哈希索引（adaptive hash index）

MyISAM

  • 支持全文索引、压缩、空间函数
  • 不支持事务和行级锁
  • 支持延迟更新索引键，可以极大地提升写入性能，但是如果主机崩溃时需要执行修复操作
  • 使用压缩表：对于不可变的数据而言可以减少I/O和节省磁盘

Memory引擎

  • 所有数据存内存
  • 可以建立hash索引
  • 读取非常非常快
  • 可以用来存储中间结果或者一些丢失了也没关系但需要快速查找的映射值

Schema与数据类型优化

选择优化数据类型

  • 更小的通常更好：选择存储数据的最小类型
  • 简单就好：尽量使用Mysql内建类型，使用Int代替字符串类型
  • 尽量避免使用NULL：包含NULL的列对于MySQL而言很难优化，使得索引等开销加大

整数类型

  • 可使用UNSIGNED进行修饰，表达范围提高一倍
  • 对于整数计算一般使用BIGINT
  • 没有必要指定整数类型的宽度，如INT(11)。这并会改变存储大小，只会影响MySQL命令行客户端的显示的字符个数

实数类型

  • FLOAT和DOUBLE类型：存储浮点数
  • DECIMAL类型：存储精确小数

字符串类型

VARCHAR
  • 用于存储可变长字符串，比定长类型更节省空间，因为它只使用必要的空间
  • 当VARCHAR需要UPDATE时，可能会使得空间增长，这样对于不同的存储引擎可能会产生碎片问题。因为空间增长时，如果达到了页的空间上限，则可能会拆分页或者拆分VARCHAR
  • 适用场景
    • 字符串列的最大长度比平均长度大很多
    • 列的更新很少，碎片不是问题
    • 使用了UTF-8这样的复杂字符集，每个字符都是用不同的字节数来存储
CHAR
  • 定长
  • 会截断末尾的空格
  • 适用场景：
    • 存储短小字符串
    • 都接近一个定长
    • 对于经常需要变更的数据，CHAR比VARCHAR好，因为不易产生碎片
    • 对于flag值或者非常短的序列，CHAR更省空间，因为不用存储变长长度

注：尽量分配小的VARCHAR，因为虽然存储空间消耗只由实际的值长度决定，但是内存消耗一般是根据VARCHAR长度的固定值来分配的。所以更长的VARCHAR会消耗更多的内存

BLOG和TEXT类型
  • 存储很大的数据，分别用于存储二进制和字符串
  • 排序时只能对每个列的最前max_sort_length进行排序
  • 不能使用全部长度字符串进行索引
枚举（ENUM）类型
  • 可以使用枚举类型来代替常用的字符串，使得空间更紧凑

日期和时间类型

  • DATETIME：能保存大范围值，从1001年到9999年，精度为秒。封装为格式为YYYYMMDDHHMMSS的整数中，和时区无关。
  • TIMESTAMP
    • 和UNIX时间戳相同，显示时间依赖于时区
    • 默认为NOT NULL
    • 通常应该是用TIMESTAMP，因为比DATETIME空间利用率高

位数据类型

  • BIT

    • MySQL默认把BIT当做字符串类型，而不是数字类型。
    • 应当尽量避免使用。如果需要一个存储true/false的值，可以使用CHAR(0)列，通过保存为NULL或者一个长度为0的字符串

  • SET

    • 保存一系列的位，可以使用MySQL的内建函数去查询
    • 可以使用TINYINT和位操作进行替代

选择标识符

  • 选择与关联表相同的标识符类型
  • 最好选择整数类型
  • 如果可能，尽量避免字符串类型
    • 因为字符串比较比整数慢
    • UUID这样随机的字符串会增加索引查找和插入性能，降低缓存命中率

MySQL schema设计中的陷阱

  • 太多的列：InnoDB列转换的消耗和列的个数成正比，避免太多的列
  • 太多的关联
  • 防止过度的使用ENUM
  • 变相ENUM：不要使用SET来代替ENUM，如set('Y', 'N')
  • 在合适的地方也可以使用NULL：比如如果确实需要一个NULL，来代表不合法的情况

范式化和反范式化

范式化的优点和缺点

  • 优点
    • 更新操作比反范式化快
    • 修改更少的数据
    • 通常表比较小，可以放到内存中，所以执行操作更快
    • 很少有重复数据，以为这很少需要使用DISTINCT或者GROUP
  • 缺点
    • 读时关联比较多

反范式化的优点和缺点

  • 优点
    • 避免关联
  • 缺点
    • 更新低效
    • 容易出现不合理的设计

缓存表和汇总表

  • 可以使用缓存表表示可以简单从schema其他表获取数据的表
  • 汇总表则保存使用GROUP BY语句聚合数据的表
